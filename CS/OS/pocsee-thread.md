# 프로세스 vs 스레드

## 프로세스
- 실행중인 프로그램 하나
- 독립된 메모리 공간 가짐(Code, Data, Heap, Stack)
- 컨텍스트 스위칭  및 생성 비용이 큼

- 예 ) 크롬, 카카오톡 인텔리제이 => 각 다른 프로세스

 ## 스레드
 - 프로세스 안에서 실행되는 작업단위
 - 같은 프로세스 메모리 공유 (Stack만 따로, 나머지는 공유)
 - 한 스레드 오류가 프로세스 전체에 영향을 줄 수 있음
 - 컨텍스트 스위칭 빠르고, 생성 비용 적음
 - 예) 크롬, 카카오톡, 인텔리제이

 ```
 프로세스
 |- 스레드 1(stack)
 |- 스레드 2(stack)
 |_ 공유 : Code, Data, Heap
 ```

 #### 스레드를 쓰는 이유
 - 프로세스 새로 만드는 건 비쌈(메모리, 시간)
 - 스레드는 메모리 공유하니까 가볍고 빠름
 - 단, 공유 메모리라서 동시성 문제 발생 가능


### 컨텍스트 스위칭
- CPU가 현재 작업을 멈추고 다른 작업으로 전환하는 것
- 프로세스 
    - 독립 메모리라 저장/복원 비용 큼
    - 주소 공간 변경까지 필요해서 더 무거움
- 스레드 : Stack 만 바꾸면 되서 전환 비용 적음

### 면접 질문
<br>
Q. 프로세스와 스레드의 차이는?
<details>
    <summary>정답</summary>
A. 프로세스는 독립된 메모리 공간을 가진 실행 단위이고,
스레드는 프로세스 안에서 메모리를 공유하며 실행되는 작업 단위이다.
스레드는 Stack만 따로 가지고 Code, Data, Heap은 공유한다. 그래서 컨텍스트 스위칭이 빠르지만 동시성 문제가 발생할 수 있다.
</details>

<br>
Q. 왜 스레드를 사용하나?
<details>
    <summary>정답</summary>
A. 프로세스 생성은 비용이 크기 때문에, 메모리를 공유하는 스레드를 사용해
컨텍스트 스위칭 비용을 줄이고 빠르게 작업을 처리할 수 있다.
단, 공유 메모리로 인한 동시성 문제가 발생할 수 있다.
</details>

### 💡 실무 연결
- Java 톰캣은 요청마다 스레드로 처리
- 여러 요청이 동시에 같은 객체 접근 가능
- 그래서 동시성 문제를 방지하기 위해 synchronized, Lock, Redis 분산락 사용
