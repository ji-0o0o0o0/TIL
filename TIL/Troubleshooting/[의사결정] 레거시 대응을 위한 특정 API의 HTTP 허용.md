## 🤔 레거시 대응을 위한 특정 API의 HTTP 허용.md

#### [상황]
이전 프로젝트 수행 시 모든 서비스를 HTTPS로 전환했으나, 특정 상담 앱 서비스만 통신이 되지 않는 이슈가 발생했습니다. 확인 결과, 해당 앱이 너무 오래되어 HTTP 통신망만 열려 있었고, 최신 보안 표준을 수용할 수 없는 상태였습니다.

#### [원인 분석: TLS 미지원과 핸드쉐이크 실패]
오래된 상담 앱이 구형 라이브러리를 사용하고 있어 최신 보안 표준인 TLS 1.2나 1.3을 해석할 수 있는 엔진이 없었던 것이 근본적인 원인이었습니다.

- SSL 핸드쉐이크 실패: 서버는 보안을 위해 HTTPS(TLS 1.2 이상)를 강제했지만, 앱이 이를 지원하지 못해 SSL 핸드쉐이크 과정에서 보안 협상에 실패하고 통신이 단절되었습니다.

#### [의사결정: 하위 호환성(Backward Compatibility) 유지]
기존 사용자의 서비스 연속성을 보장하는 것이 최우선이라고 판단했습니다. 보안상 취약점은 인지하되, 하위 호환성을 유지하기 위해 해당 API에 한해서만 HTTP 통신을 허용하는 차선책을 선택했습니다.

#### [배운 점]
기술적으로 더 우수한 방식(HTTPS)이 있더라도, 실무에서는 레거시 시스템과의 연결성이나 인프라 제약 사항에 따라 최선의 절충안을 찾아야 한다는 것을 배웠습니다. 이를 통해 시스템 설계 시 신규 기능 개발만큼이나 하위 호환성 전략이 중요하다는 것을 체감했습니다.

#### 용어 정리
1. **하위 호환성(Backward Compatibility)**: 새로운 시스템이 옛날 시스템이나 데이터를 여전히 지원해주는 것.
2. **TLS(Transport Layer Security)**: 인터넷 데이터 암호화 표준 규칙. HTTPS의 'S'를 가능하게 하며, SSL의 개선된 버전. (현재 TLS 1.2 이상 권장)
3. **SSL 핸드쉐이크(SSL Handshake)**: 본 통신 전 클라이언트와 서버가 암호화 방식(TLS 버전 등)을 협상하는 과정.
